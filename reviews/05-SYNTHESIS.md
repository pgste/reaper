# Reaper — Enterprise Readiness Review: Synthesis

**Overall verdict: NOT READY.**

Three of four reviewers returned **NOT READY** (Security, Code & API, Product Architecture); Performance returned **CONDITIONAL**. Worst-persona-wins applies and is not overridden — the NOT READY verdicts rest on **multiple independent P0s**, and the two most severe were found *independently by two reviewers citing the same `file:line`*, which raises them from "assertion" to "verified." Reaper is authorization infrastructure; the blockers below let an anonymous network attacker control live authorization decisions and rewrite policy for any tenant.

> **Review base:** conducted against commit `8abb3b8` (working tree at review time), which is ~2 merged PRs / 729 lines behind `main`. That delta is CI config, the string-interner memory reclamation, and docker-compose fixes — none of it touches any P0/P1 below. Findings are architectural and hold against `main`.

## Executive summary (≤10 lines)
1. **The enforcement plane has no authentication.** The agent (the component that decides allow/deny) serves policy-deploy, entity/relationship writes, and audit dumps on `0.0.0.0` with TLS off and **zero auth** (Security P0-1 = Code API-2, same `main.rs:490-539`).
2. **The control plane has no authentication on whole route groups** — bundles, policies, orgs, teams, billing — so an anonymous caller does cross-tenant policy CRUD and **promotion to agents**, with IDOR on global bundle UUIDs (Security P0-3/P0-3b = Code API-1).
3. **One unauthenticated POST bypasses the entire bundle-signing chain** — the direct deploy path never verifies signatures, while the pull path does (Security P0-2).
4. **No SSO/SAML/OIDC and no SCIM** for the human admin console — a hard gate on any bank security questionnaire (Product F1).
5. **The audit trail is not defensible to a regulator** — not tamper-evident, lossy under load, and disable-able by config, so "prove decision X used policy vX at time T" cannot be guaranteed (Security P1-2).
6. **Distribution is the weakest subsystem by consensus:** signing bypass, no anti-rollback/revocation, the "policy-as-code" pillar is wired only in a test and never runs in `main.rs`, and the Git link is user-token/broad-scope/poll-only.
7. **Availability:** `panic = "abort"` + no catch layer + an unbounded-recursion DSL and a synchronous batch loop on the async runtime mean a crafted policy or batch can abort a sidecar (Code API-3, Security P2-1, Perf P1-2).
8. **No supply-chain assurance** — no `cargo audit`/`cargo deny`/SBOM/fuzz in CI (converged by 3 reviewers).
9. The cryptographic and correctness *primitives* are good (signing, JWKS, Argon2, compiled DSL hot path, agent-confirmed rollout convergence, delta-sync). **The failures are missing wiring, not missing knowledge** — which is fixable, but today they are disqualifying.
10. Net: strong engine + distribution machinery (~70% of a product) wrapped in a **~30% enterprise control plane** whose auth is opt-in and fails open.

## Cross-cutting themes (where reviewers converged independently)

1. **Authorization is opt-in and fails open.** The single dominant theme. Security and Code independently found the same two unauthenticated surfaces (agent + control plane) at the same lines. Auth exists as a *per-handler extractor* (`RequireAuth`) that some files simply forgot — there is no default-deny gateway. This is the #1 reason for the verdict.
2. **The distribution / promotion path is the weakest subsystem.** All three non-perf reviewers flag it: signing bypass (Sec), no anti-rollback (Sec), GitOps not wired + wrong-shape git link (Product), no idempotency on promote (Code API-14). This is where to concentrate hardening.
3. **No software supply-chain assurance.** Three reviewers independently flagged the absence of `cargo audit`/`cargo deny`/SBOM/fuzz. Cheap to fix, expected by every regulated buyer.
4. **The audit substrate can't yet meet its own compliance promise.** Good *attribution* (policy version, data checksum) undermined by no *integrity* (tamper-evidence) and no *completeness* guarantee (lossy + samplable). Replay is reproduction-only, not counterfactual.
5. **Availability under hostile input is unproven.** `panic = "abort"` turns any reachable panic into an outage; the DSL is not total/terminating by construction; batch and ReBAC traversal multipliers are per-request-unbounded.

## Adjudication of disagreements / nuance
- **Perf (CONDITIONAL) vs the rest (NOT READY):** not a conflict — Perf correctly scoped to performance and correctly found no *perf* P0. The overall gate is worst-persona; NOT READY stands.
- **The "~957 unwrap/expect/panic" headline is overstated as a raw number.** Code noted the count is inflated by inline `#[cfg(test)]` modules and that hot-path unwraps are guarded; Security independently confirmed no unguarded unwrap on request-derived data in the eval hot path. The *real* availability risk is narrower and sharper: `panic = "abort"`, unbounded DSL recursion, and specific reachable panics (e.g. the `Content-Disposition` header-builder `unwrap` on an attacker-controlled bundle name, Code API-11). Fix those; don't be reassured by "hot path is guarded."
- **Signing: simultaneously "done well" and a P0.** Both are true and must be stated together — the *primitive* and the *pull path* are correct and fail-closed (`require_signed_bundles = true`), which is exactly why the *unverified push path* (P0-2) is so dangerous: it silently routes around a control everyone assumes is on.

## If a UK bank's architecture review board saw this tomorrow, the first three rejections
**First, the identity gap on the security questionnaire:** the enforcement agent and whole control-plane route groups accept unauthenticated requests (an anonymous actor can change live authorization decisions and promote policy for any tenant), and the admin console has no SSO/SAML/OIDC or SCIM — so privileged actions cannot be tied to corporate identity or deprovisioned on termination. That alone is a hard stop. **Second, the audit and integrity story fails the regulator's core question:** decision logs are not tamper-evident, are lossy, and can be disabled, and the signing chain that protects policy distribution is bypassable via an unverified deploy path with no anti-rollback — so "prove which policy version made this decision, and prove it wasn't tampered with" cannot be answered under DORA/SOC 2. **Third, operational resilience and supply-chain governance are undocumented:** no control-plane HA/DR/backup or RPO/RTO, `panic = "abort"` making a crafted policy a denial-of-service against the sidecar, and no dependency scanning/SBOM — which the board's operational-resilience (SS2/21, DORA) and third-party-risk reviewers will each reject independently.

**The good news for a remediation plan:** every P0 is missing *wiring* around correct primitives, not a missing capability. A default-deny auth gateway on both planes, routing all bundle loads through the existing `verify_bundle`, spawning the existing `SyncService`, adding SSO/SCIM, and hash-chaining the existing decision records would move the verdict from NOT READY toward CONDITIONAL without new subsystems. Sequence: **(1) close the auth P0s on agent + control plane; (2) route every deploy path through signature verification; (3) SSO/SCIM; (4) tamper-evident + mandatory-audit mode; (5) supply-chain gates.**
